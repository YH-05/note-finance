# X投稿生成スキル

note記事ディレクトリから、フックの利いた280字以内のX投稿文を生成する。

## 入力

| パラメータ | 必須 | 説明 |
|-----------|------|------|
| @article_dir | ○ | 記事ディレクトリ（`articles/{category}/{YYYY-MM-DD}_{slug}/`） |

引数なしで呼ばれた場合は、カレント会話で直前に扱っていた記事ディレクトリを使う。
それも不明な場合はユーザーに記事ディレクトリを質問する。

```
/x-post @articles/asset_management/2026-03-15_investment-psychology-behavioral-bias/
```

## 出力

`{article_dir}/02_draft/x_post.md`

```markdown
---
article_id: {article_id}
category: {category}
platform: x
char_limit: 280
char_count: {実際の文字数}
status: draft
---

{投稿本文}
```

## 処理フロー

### Step 1: 記事コンテンツの読み込み

優先順位に従って記事本文を読む:
1. `02_draft/revised_draft.md`（最優先）
2. `02_draft/first_draft.md`（フォールバック）

`meta.yaml` からカテゴリ・トピック・target_audience を取得。

### Step 2: 記事の核心を抽出

記事から以下を特定する:
- **タイトル・テーマ**: 読者が得る最大の価値は何か
- **フックになる事実/逆説**: 意外性のある数字、「常識」への反論、感情的共鳴ポイント
- **記事の核心3点**: 読者が「続きを読みたい」と思うポイント
- **具体的な数字**: あれば必ず使う（例: 「2倍」「7%」「年4.3万円」）

### Step 3: 書き出しパターンの選択

記事の特性に応じて最も刺さるパターンを1つ選ぶ:

**パターンA: 逆説型**（感情・心理系の記事に適する）
```
{通説・思い込み}は「{一般的な原因}」ではありません。

{本当の理由}だからです。
```
例: 「暴落で売りたくなるのは『意志が弱いから』ではありません。」

**パターンB: 疑問形型**（「常識を検証する」系の記事に適する）
```
「{よくある思い込みや問い}」は本当？
```
例: 「「S&P500だけ買えばOK」は本当？」

**パターンC: 数字型**（制度・シミュレーション系の記事に適する）
```
{名詞/制度}、{変化の数字}。
```
例: 「会社員のiDeCo上限、2027年に月1.2万→2万円へ引き上げ予定。」

### Step 4: 投稿文の構成

以下のテンプレートに沿って構成する:

```
{フック（パターンA/B/Cの書き出し）}

{記事の核心ポイント 2〜3点（箇条書き）}

{誘導文（「解説しました」「詳しくはnote記事で」など1文）}
{URL}

{ハッシュタグ 3〜5個}
```

**構成上の制約:**
- 全体で **280字以内**（URLは23字として計算しない ← Twitterのt.co短縮を想定）
- 箇条書きは `・` または `-` で統一（絵文字は読者層・トーンに合わせて判断）
- ハッシュタグ間はスペース区切り
- `{URL}` はプレースホルダーのままにする（実URLは投稿時に差し替え）

### Step 5: ハッシュタグ選択

カテゴリ別デフォルトタグ + 記事内容から2-3個追加:

| カテゴリ | デフォルトハッシュタグ |
|---------|----------------------|
| asset_formation | #資産形成 #投資初心者 #新NISA |
| asset_management | #資産形成 #投資初心者 #NISA |
| stock_analysis | #株式投資 #株 #投資 |
| macro_economy | #マクロ経済 #経済 #投資 |
| weekly_report | #米国株 #週間レポート #投資 |
| quant_analysis | #クオンツ #投資戦略 #データ分析 |
| side_business | #副業 #資産形成 |

記事キーワードが `iDeCo` → `#iDeCo #節税`、`行動経済学` → `#行動経済学`、
`インデックス` → `#インデックス投資`、`債券` → `#債券投資` のように追加。

### Step 6: 文字数チェックと調整

実際に文字数をカウントする（`{URL}` は23字として計算）。

**280字超の場合の削減優先順位:**
1. 箇条書きの説明を短縮
2. 箇条書きを3点→2点に削減
3. フックの文章を短縮
4. ハッシュタグを1-2個削減（ただし最低3個は残す）

### Step 7: ファイル出力

`02_draft/x_post.md` を以下の形式で出力:

```markdown
---
article_id: {meta.yamlのarticle_id}
category: {meta.yamlのcategory}
platform: x
char_limit: 280
char_count: {実際の文字数（URLを23字計算）}
status: draft
---

{投稿本文}
```

既存の `x_post.md` がある場合は上書きではなく内容を確認し、
ユーザーに「既存ファイルを上書きしますか？」と確認する。

## 完了報告

```
## X投稿文を生成しました

- **文字数**: {char_count}字 / 280字
- **カテゴリ**: {category}
- **書き出しパターン**: {A/B/C型}
- **ハッシュタグ**: {tags}
- **出力ファイル**: {article_dir}/02_draft/x_post.md

投稿前に `{URL}` を実際のnote記事URLに置き換えてください。
```
