## 層別差分の詳細

### beginner 専用ルール

**禁止専門用語（括弧解説なしで使用禁止）:**
- ROE → 「自己資本利益率（ROE）」または「利益率」
- DCF → 「企業価値計算」
- ボラティリティ → 「値動きの激しさ」
- リバランス → 「定期的な比率調整（リバランス）」
- スプレッド → 「利回りの差」
- コンセンサス → 「アナリストの予想平均」
- EPS → 「1株あたり利益（EPS）」
- PBR/PER → 「株価指標（PBR/PER）」

**感情軸の選び方:**
- 「自分でもできる」という安心感 → 達成可能なステップを提示
- 「知らなかった」という発見 → 意外な数字・事実を冒頭に
- 「損しているかも」という危機感 → 制度変更・機会損失の数字

**CTA例:**
- 「解説しました！」
- 「まとめました！」
- 「詳しくはnoteで🎯」

### intermediate 専用ルール

**推奨専門用語（解説不要）:**
- S&P500, MSCI, 日経225
- インデックスファンド, ETF
- ポートフォリオ, アセットアロケーション
- 利回り, 配当利回り
- PER, PBR（投資経験者向け一般用語）

**感情軸の選び方:**
- 「実は損してた」という気づき → 手数料・機会損失の比較
- 「自分の戦略を見直したい」 → データによる通説の検証
- 「このタイミングで動くべき？」 → 転換点・マクロ変化の宣言

**CTA例:**
- 「解説しました。」
- 「検証しました。」
- 「整理しました。」

### advanced 専用ルール

**必須要素（含まないと信頼感が下がる）:**
- 具体的な数値・指標（例: 「FCFイールド4.2%」「-23%」）
- 時間軸・サンプル数（例: 「過去4サイクル」「3回中2回」）
- 因果関係のロジック（例: 「〜のため〜が先行する」）

**避けるべき表現:**
- 「分かりやすく解説」→ 読者に失礼
- 「初心者でも」→ 誤ったターゲティング
- 感嘆符全般
- 過剰な絵文字

**CTA例:**
- 「を検証しました。」
- 「を分析しました。」
- 「詳細はnoteで。」

## フックパターン選択の判断基準

### パターンA（逆説型）が最適なケース
- 記事テーマが「心理・行動」系
- 「〜と思われているが、実は〜」という構造
- 読者の自己否定感を解放できる内容

### パターンB（疑問形型）が最適なケース
- 一般的な「正解」とされる主張を検証する記事
- 比較・ランキング・「どちらが良い？」系
- データ・根拠で通説を揺さぶる内容

### パターンC（数字型）が最適なケース
- 制度変更・法改正の記事
- シミュレーション・試算の記事
- 統計・データを核心にした記事

### パターンD（宣言型）が最適なケース
- マクロの転換点・重要局面を告げる記事
- 「逆張り」的な見解を提示する記事
- 上級者向けの判断・評価を含む記事

## よくある失敗パターンと対策

### 失敗1: フックが弱い
**症状**: 最初の一文が「〜について解説しました」で始まる
**対策**: フック（A/B/C/D）から必ず始め、CTA（誘導文）を最後に持ってくる

### 失敗2: 専門用語の出し惜しみ（intermediate/advanced向け）
**症状**: 「投資の仕組み」「リスク管理」などふわっとした言葉のみ
**対策**: 具体的な指標名・数値・制度名を積極的に使う

### 失敗3: 情報が多すぎる
**症状**: 箇条書きが4点以上、280字超え
**対策**: 最も刺さる2点に絞る。「記事で残り〇点を解説」というCTAに変換

### 失敗4: 層ミスマッチ
**症状**: beginner向けの絵文字・感嘆符が advanced 記事に入る
**対策**: Step 0 でaudience を確定させてから文体を統一する

### 失敗5: URLプレースホルダーを実URLに書き換える
**症状**: `{URL}` が `https://note.com/...` に書き換えられている
**対策**: `{URL}` は常にプレースホルダーのまま残す

## 参照ライブラリの活用方法

### 参照サンプルを使ってできること
1. `why_it_works` から「なぜ刺さるのか」を学んで同様のロジックを適用
2. 文字数・構成のバランスを参考にする
3. ハッシュタグの選び方・数を参考にする
4. 絵文字の使い方（層による差）を参考にする

### 参照ライブラリを更新する時期
- X に投稿した後、エンゲージメントが取れたら `engagement_score` を更新
- 新しいカテゴリ・層の記事を初めて作る際にサンプルを追加
- 既存サンプルより明らかに良い投稿ができたら置き換えを検討

## 特殊ケースの対応

### `meta.yaml` がない場合
- 記事ディレクトリの構造からカテゴリを推測（ディレクトリ名の第1セグメント）
- audience は `intermediate` をデフォルト

### `revised_draft.md` も `first_draft.md` もない場合
- ユーザーに「記事ドラフトがありません。ファイルを確認してください」と伝える

### 記事の内容が複数のカテゴリにまたがる場合
- 主要な読者層が一番メリットを受けるカテゴリを選択
- `meta.yaml` の `category` フィールドを優先

### 週次レポートの場合
- `weekly_report` カテゴリとして扱う
- audience は基本的に `intermediate`（weekly_report に beginner は想定しない）
- 数字が多いため C型または D型を推奨
