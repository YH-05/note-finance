# 議論メモ: creator-neo4j オントロジーファースト再設計

**日付**: 2026-03-22
**参加**: ユーザー + AI

## 背景・コンテキスト

creator-enrichment セッション（4サイクル）と定量分析を経て、creator-neo4j の根本課題を特定。
Entity 未接続率84%や RELATES_TO 25件という表層的な問題の背後にある構造的欠陥は
**オントロジー（構成原則）の欠如**であるとユーザーが指摘。

現状の汎用 entity_type（platform, occupation, technique 等）では、
同じ「platform」でも集客チャネル（Instagram）と販売マーケットプレイス（ココナラ）の
役割の違いを表現できず、データを増やしても接続されない。

## 議論の要点

### 1. オントロジーファーストの合意

- データ投入よりもドメイン概念定義を先行させる
- Entity 抽出は全コンテンツで必須化
- Entity リンキングでオントロジーに接続
- トリプル抽出は数値データ・因果関係など必要な場合に限定

### 2. トリプル vs Entity リンキングの整理

- リレーションの MERGE はキーに述語（rel_detail）を含めれば同一ペアで複数リレーション可能
- Claim ノードは「同一事実を複数ソースが裏付ける」場合に必要
- ハッシュによる claim_key は不要 → 主語・述語・目的語の MERGE で同一性は担保される
- 全文トリプル化はコスト過大 → オントロジー準拠の Entity リンキングで接続性を確保

### 3. 6フェーズの工程表

| Phase | 内容 | タスク数 |
|-------|------|---------|
| A 設計 | 目的定義→オントロジー→スキーマ→正規化ルール | 4 |
| B パイプライン実装 | 抽出プロンプト→リンキング→emit改修→save改修→トリプル | 5 |
| C 既存データ移行 | Entity再分類→後付け抽出→authority_level→旧スキーマ整理 | 4 |
| D 品質保証 | 適合検証→重複検出→孤立検出→カバレッジ計測 | 4 |
| E 運用改修 | enrichmentスキル→ギャップ分析→Phase 4.5 | 3 |
| F 活用 | 素材クエリ→パターン発見→記事フロー統合 | 3 |

依存: A→B→C→D は順序必須、E・F は D と並行可能

## 決定事項

1. **オントロジーファースト**: データ投入よりドメイン概念定義を先行
2. **6フェーズ23タスク**: A設計→Bパイプライン→C移行→D品質→E運用→F活用
3. **トリプル抽出は限定的**: Entity リンキングを主軸とし、トリプルは構造化が必要な事実のみ
